# 블로그 글 생성 파이프라인

## 입력 수집 (글쓰기 전에 반드시 읽는다)

1. `PLAN.md` — 문제, 타겟, 핵심 기능
2. `BUILD_LOG.md` — Decisions, Stuck & Solved, Ship 섹션
3. `git log --oneline` — 작업 흐름 파악
4. 프로덕션 URL

재료가 부족하면(Decisions 2개 미만, Stuck & Solved 0개) 글을 쓰지 말고 사용자에게 부족한 섹션을 알려라. 재료 없는 글은 일반론이 되고, 일반론은 내 글이 아니다.

## 글의 포지셔닝: 개발 글이 본체, 홍보는 구조로 해결

원칙: **독자가 "개발 글 잘 읽었는데 제품도 궁금해지네"가 되어야지, "홍보 글이네"가 되면 실패다.** 홍보를 문장으로 하지 말고 구조로 한다 — 제품 노출 위치를 고정하고, 본문은 순수하게 기술적 가치를 준다.

분량 비율 가이드: 제품 소개 ~15% / 개발 내용 ~70% / 결과·CTA ~15%

## 글 구조 (고정 템플릿)

```
1. 훅 (2~3문장)
   타겟이 겪는 문제 상황으로 시작. 제품 얘기 아님.

2. 만든 것 (1문단 + 링크 1회)
   "그래서 <제품명>을 만들었다" + 한 줄 설명 + URL.
   스크린샷 자리 표시: 아래 "이미지" 규칙의 업로드 마커를 넣는다.

3. 기술적 결정 (본문의 심장, 1~2개만)
   BUILD_LOG의 Decisions에서 가장 흥미로운 것 1~2개를 골라
   '대안이 있었고, 이런 이유로 이걸 골랐고, 트레이드오프는 이것' 서사로 푼다.
   필요하면 코드 블록 1~2개 (각 15줄 이하).

4. 막혔던 지점 (Stuck & Solved에서 1개)
   증상 → 삽질 → 원인 → 해결. 검색 유입의 핵심이고 신뢰의 원천.
   같은 에러를 겪을 사람이 검색할 키워드를 소제목에 넣는다.

5. 결과 (숫자, 솔직하게)
   걸린 시간, 코드 규모, 배포 후 초기 지표.
   결제 0건이면 0건이라고 쓴다 — 실패 회차의 솔직함이 채널의 차별점이다.

6. 마무리 + CTA (1개만, 링크 2회째이자 마지막)
   "△△가 필요했던 분이라면 <URL>에서 써보고 피드백 주시면 반영하겠다"
   + 다음 빌드 예고 한 줄.
```

## 홍보 규칙 (어기면 광고 글이 된다)

- 제품 링크는 정확히 2회: 섹션 2와 섹션 6. 본문 중간 삽입 금지.
- CTA는 1개만. "써보세요 + 구독하세요 + 공유하세요" 같은 다중 요청 금지.
- 과장 형용사 금지: "혁신적인", "강력한", "놀라운" → 전부 구체적 사실로 대체 ("3초 걸리던 게 0.4초").
- 경쟁 서비스 비방 금지. 비교가 필요하면 사실만.
- 타겟 독자의 언어로 쓴다. PLAN.md의 타겟이 비개발자면 섹션 3~4의 기술 깊이를 낮추고 "어떻게 동작하나" 수준으로.

## 이미지 (상대경로는 깨지지만 공개 절대 URL은 된다)

핵심 구분: 티스토리는 마크다운의 **상대경로 이미지(`![](./screenshots/x.png)`)는 못 읽지만**, **공개 절대 URL(`![](https://...)`)은 그대로 렌더링**한다. 그러니 로컬 경로만 피하면 드래그 없이 자동으로 이미지가 뜨게 만들 수 있다. (티스토리 Open API는 2024.2 종료돼 이미지 업로드 API 경로는 없다.)

우선순위대로:

1. **GitHub raw 절대 URL (기본 — repo가 public일 때).** 스크린샷을 repo에 commit/push한 뒤, 본문에 raw URL을 임베드한다:
   ```
   ![<캡션>](https://raw.githubusercontent.com/<owner>/<repo>/<기본브랜치>/posts/screenshots/<파일>.png)
   ```
   - 전제: 그 파일이 해당 브랜치에 push 돼 있어야 URL이 산다. push 전제를 HTML 주석으로 본문에 남긴다.
   - private repo면 raw URL이 인증을 요구해 안 뜬다 → 2번으로.
2. **외부 이미지 호스트 절대 URL.** repo가 private이거나 GitHub에 두기 싫으면 공개 CDN/버킷(Cloudflare R2, S3, imgur 등)에 올리고 그 URL을 임베드한다.
3. **업로드 마커 (위 둘 다 안 될 때 폴백).** 사람이 편집기에서 직접 올려야 하면 마커로 위치를 명시한다:
   ```
   > 📷 **[이미지 업로드 위치]** 편집기에서 이 자리에 `posts/screenshots/<파일>.png`를 직접 드래그해 올린다. 캡션: *<한 줄 캡션>*
   ```

공통 규칙:

- 이미지는 1~2장까지만 (섹션 2의 핵심 화면 1장이 기본). 본문은 스크린샷 없이도 이해되게 쓴다.
- `![](상대경로)`(`./`, `../`로 시작) 임베드는 절대 쓰지 않는다 — 게시 시 깨진다.

## 문체

`blog-style` 스킬(프로젝트 또는 ~/.claude/skills/blog-style)의 규칙을 반드시 적용한다.

## 출력물 (바로 게시 가능한 완성본)

초안이 아니라 그대로 복붙해 게시할 수 있는 형태로 만든다. 후보를 나열하거나 선택을 사용자에게 미루지 않는다.

1. 제목 1개 — 글 내용에 맞춰 검색 키워드형/서사형 중 하나를 골라 확정한다. 후보 나열 금지.
2. 본문 (마크다운, 1,500~2,500자) — 맨 위에 위 제목을 H1(`# 제목`)으로 넣어 게시 가능한 완성 형태로.
3. 태그 5~7개 (기술 태그 + 제품 도메인 태그 혼합)
4. 보너스: 쓰레드/X용 5~7문장 요약 (훅 1문장 + 핵심 결정 1개 + 숫자 + 링크), 링크드인용 1문단

## 셀프 체크 (출력 전 검증)

- [ ] 제목이 1개로 확정돼 본문 맨 위 H1로 들어갔는가 (후보 나열 없음)
- [ ] 이미지는 상대경로 임베드 없이 공개 절대 URL(GitHub raw 등) 또는 업로드 마커로 넣었는가 (티스토리에서 안 깨지게)
- [ ] 제품 링크 2회, CTA 1개인가
- [ ] 과장 형용사가 없는가
- [ ] 코드 블록이 각 15줄 이하인가
- [ ] Stuck & Solved 소제목에 검색 가능한 키워드가 들어갔는가
- [ ] 숫자가 최소 2개 들어갔는가 (시간, 지표, 성능 등)
- [ ] blog-style 셀프 체크를 통과했는가
