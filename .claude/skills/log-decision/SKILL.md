# 의사결정 기록 (BUILD_LOG.md)

이 로그는 두 가지 용도다: (1) 미래의 나/에이전트가 "왜 이렇게 했지?"를 추적, (2) **`write-post` 스킬이 블로그 글을 생성할 때의 원재료.** 형식이 고정돼 있어야 자동화 품질이 유지되므로 형식을 임의로 바꾸지 않는다.

## 기록 형식

BUILD_LOG.md의 `## Decisions` 섹션에 한 결정당 한 블록:

```markdown
### [D1 14:20] Supabase Auth 대신 매직링크 자체 구현 안 함

- 선택: Supabase Auth 매직링크
- 대안: NextAuth, 자체 구현
- 이유: 이번 빌드에서 인증은 차별화 요소가 아님. 검증된 것을 쓰고 핵심 기능에 시간 투자
- 트레이드오프: 벤더 종속. 단 MVP 단계에서 무의미한 우려
```

규칙:

- 시각은 대략으로 충분 (`[D1 14:20]` = 빌드 1일차)
- 4줄(선택/대안/이유/트레이드오프)을 넘기지 않는다 — 길게 쓰고 싶은 만큼 중요한 결정이면 docs/adr/에 ADR로 쓰고 여기엔 링크만
- "이유"에는 일반론이 아니라 **이번 빌드의 맥락**을 쓴다 ("이번 범위에선", "이 타겟에게는" 등)

## BUILD_LOG.md 전체 구조 (고정)

```markdown
# Build Log — <제품명>

## Decisions

(위 형식의 블록들)

## Stuck & Solved

(막혔던 지점: 증상 한 줄 → 원인 한 줄 → 해결 한 줄. 블로그 글의 가장 좋은 재료)

## Backlog

(PLAN.md 밖에서 발견한 개선점. 구현하지 않고 여기에만 적는다)

## Ship

(배포 시각, URL, 검증 지표, 초기 숫자)

## Retro

(배포 직후 3줄 회고: 잘된 것 / 안된 것 / 다음 빌드에 바꿀 것)
```

## 기록 트리거 (이럴 때 반드시 기록)

- 라이브러리/서비스를 선택했을 때 (대안이 존재했다면 무조건)
- 기본 스택에서 벗어났을 때
- 30분 이상 막혔다가 해결했을 때 → `## Stuck & Solved`
- PLAN.md 범위를 자르거나 바꿨을 때
