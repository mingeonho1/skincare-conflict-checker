# 같이써도돼? — 스킨케어 성분 충돌 체커

## 문제

레티놀·비타민C·산(AHA/BHA) 같은 액티브 화장품을 막 쓰기 시작한 사람이 "이거 같이 발라도 되나?"를 검색해도, 출처가 제각각이고 과장된 통념(나이아신아마이드+비타민C 위험설 등)까지 섞여 있어 뭘 믿어야 할지 모른 채 불안하거나 무작정 다 따로 바른다.

## 타겟

전성분에 레티놀·비타민C·살리실산이 적힌 세럼을 두세 개 들였는데 "동시에 바르면 자극 난다던데?"가 불안한 25~35세 스킨케어 입문자 — 특히 트러블이 났는데 그게 조합 탓인지 모르는 사람.

## 핵심 기능 (단 1개)

내 화장대에 있는 제품 2~6개의 전성분을 붙여넣은 사용자가 "충돌 검사하기"를 누르면 → ① 함께 쓰면 안 되는 성분 쌍을 메커니즘·심각도·대처법과 함께 경고받고, ② "이건 같이 써도 괜찮아요" 안심 카드로 통념을 바로잡고, ③ 안전한 아침/저녁 사용 순서 타임라인을 얻는다.

## 깊이 명세 (이 1개를 상용급으로 — 의무)

### 아키텍처 원칙 (이번 빌드의 가장 중요한 제약 — 위반 불가)

- **충돌 '판정'은 규칙 기반(결정론적)이 유일한 진실의 원천이다.** 큐레이션된 액티브 성분 사전(약 10종)과 충돌 규칙 테이블을 코드(`schema.ts` + 정적 데이터)로 직접 구축한다. zod로 규칙·결과 스키마를 검증한다.
- **Gemini(LLM)는 두 가지에만 쓴다**: (a) 입력된 전성분 텍스트에서 위 사전의 액티브 성분이 들어있는지 **추출/정규화**(한글·영문·INCI 별칭 매핑), (b) 이미 규칙 엔진이 내린 충돌 결과를 사용자 친화적 한국어 설명으로 **윤문**. 충돌 '여부 판정'은 절대 LLM에 맡기지 않는다(LLM은 안전정보 인용을 환각함).
- **Gemini 미설정/실패/타임아웃 시에도 규칙 엔진은 항상 동작한다.** 추출은 사전 별칭 키워드 매칭으로 폴백, 설명은 룰테이블에 내장된 기본 한국어 카피로 폴백한다. 즉 핵심 기능은 LLM 없이도 100% 작동하고, LLM은 정확도·자연스러움을 더하는 부가 레이어다.
- `GEMINI_API_KEY`는 `src/lib/env.ts`에서 zod로 옵셔널 검증한다. 미설정이면 폴백 경로로 분기(에러 아님).

### 큐레이션 액티브 성분 사전 (코드 데이터 — 이대로 구현)

각 성분: `id` / `displayName`(한국어) / `aliases`(매칭용 한글·영문·INCI 별칭 배열) / `category` / `photosensitive`(광민감 여부) / `timeOfDay`(권장 시간대 'AM'|'PM'|'ANY') / `viscosityRank`(레이어링 순서용 정수, 묽을수록 작음).

1. 레티놀/레티노이드 — 별칭: 레티놀, 레티날, 레티놀데하이드, 레티닐팔미테이트, retinol, retinal, retinaldehyde, retinyl palmitate, adapalene, 아다팔렌 / PM / 광민감
2. AHA(글라이콜릭산) — 별칭: 글라이콜릭애씨드, 글리콜산, glycolic acid / PM / 광민감
3. AHA(락틱산) — 별칭: 락틱애씨드, 젖산, lactic acid / PM / 광민감
4. BHA(살리실산) — 별칭: 살리실릭애씨드, 살리실산, salicylic acid, BHA, beta hydroxy acid / PM / 광민감
5. 비타민C(아스코르빅) — 별칭: 아스코빅애씨드, 아스코르브산, ascorbic acid, l-ascorbic acid, 에틸아스코빌에텔, sodium ascorbyl phosphate, 비타민C / AM
6. 벤조일퍼옥사이드 — 별칭: 벤조일퍼옥사이드, benzoyl peroxide, BPO / ANY
7. 나이아신아마이드 — 별칭: 나이아신아마이드, 니아신아마이드, niacinamide, nicotinamide / ANY
8. 펩타이드 — 별칭: 펩타이드, peptide, 팔미토일펜타펩타이드, copper peptide, 구리펩타이드 / ANY
9. (보조 안심용) 히알루론산 — 별칭: 히알루로닉애씨드, 소듐하이알루로네이트, hyaluronic acid, sodium hyaluronate / ANY (묽은 수분, 항상 안전)
10. (시간대 앵커) 자외선차단 — 별칭: 자외선차단, 선크림, SPF, 옥토크릴렌, 징크옥사이드, zinc oxide, titanium dioxide, 티타늄디옥사이드 / AM (감지되면 광민감 보너스 안내에 사용)

### 충돌 규칙 테이블 (코드 데이터 — 출처 교차검증됨, 이대로 구현)

각 규칙: `pair`(성분 id 쌍) / `verdict`('warn'|'safe') / `severity`('high'|'moderate'|null) / `mechanism`(왜 충돌/안전한지 기본 한국어) / `action`(대처법 기본 한국어).

**경고(warn) 규칙:**

- 레티놀/레티노이드 × AHA(글라이콜릭) / AHA(락틱) / BHA(살리실산) → severity: high. 메커니즘: 장벽 손상·자극·광민감 증가. 대처: 같은 시간대 동시 사용을 피하고 격일 또는 아침/저녁 분리.
- 비타민C(아스코르빅) × 벤조일퍼옥사이드 → severity: high. 메커니즘: BP 산화로 비타민C가 분해되어 효과 저하 + 자극. 대처: 비타민C는 아침, BP는 저녁 또는 격일.
- 레티놀/레티노이드 × 벤조일퍼옥사이드 → severity: moderate(보수적). 메커니즘: 동시 사용 시 레티놀 불활성화 가능. 대처: 시간대 분리 권장.

**안심(safe) 규칙 — 명시적으로 초록 카드로 안심시킨다(경고 금지):**

- 나이아신아마이드 × 비타민C → safe. 메커니즘: 함께 써도 안전한 조합입니다(과거 위험설은 반박됨). 시너지 가능.
- 나이아신아마이드 × 레티놀 → safe. 메커니즘: 안전하고 시너지(나이아신아마이드가 장벽을 도와 레티놀 자극을 완화). 순서 권장: 나이아신아마이드 먼저, 레티놀 나중.
- 레티놀 × 비타민C → safe. 메커니즘: "pH 충돌"설은 반박됨. 시간대만 나누면(C 아침/레티놀 저녁) 무리 없습니다.

**규칙 우선순위**: 동일 성분 쌍에 warn과 safe가 둘 다 매칭될 일은 없도록 테이블을 설계하되, 매칭되지 않은 쌍은 "특별한 충돌 정보 없음"으로 침묵한다(과잉 경고 금지 — 이게 신뢰 차별화). safe 규칙이 있는 쌍만 초록 카드로 노출하고, 그 외 무규칙 쌍은 카드를 만들지 않는다.

### AM/PM 순서 산출 원칙 (결정론적)

- 감지된 성분을 AM 그룹 / PM 그룹으로 배치: 광민감(레티놀·AHA·BHA)은 PM, 비타민C·자외선차단은 AM, ANY는 양쪽 중 충돌이 적은 쪽(기본 AM, 단 PM에 같은 성분의 충돌상대가 있으면 분리).
- 각 그룹 내부는 `viscosityRank` 오름차순(묽은 토너/에센스 → 진한 세럼 → 크림)으로 정렬.
- 충돌 쌍은 가능하면 AM/PM으로 갈라 배치한다(예: 비타민C↔BP를 AM/PM 분리).
- 출력 카피는 "권장 순서"로 단정하지 않는다("절대 규칙은 아니에요" 톤).

### 엣지케이스·실패경로 (보이는 것 + 복구)

1. **전성분이 비었거나 30자 미만** → 해당 제품 카드에 인라인 "전성분을 더 붙여넣어 주세요" 경고, 검사 버튼은 활성이되 그 카드는 "미입력"으로 표시. 검사 결과에서 "N번 제품은 성분이 너무 짧아 분석에서 제외했어요" 안내.
2. **액티브 성분이 하나도 안 잡힘(전 제품)** → 결과 화면에서 빨간/초록 카드 대신 중립 카드: "충돌을 일으킬 만한 액티브 성분(레티놀·산·비타민C 등)이 감지되지 않았어요. 함께 써도 무난합니다." + 우리가 추적하는 성분 목록을 펼쳐보기로 제시.
3. **제품 1개만 유효 입력** → 검사는 가능하나 "비교할 제품이 1개라 충돌 검사는 제품을 2개 이상 넣어야 해요" 안내 + 그 1개 제품 내부에 다중 액티브가 있으면 단일 제품 내 충돌도 검사해 보여준다.
4. **Gemini 응답 실패/타임아웃(8초 초과)** → 키워드 매칭 폴백으로 즉시 전환, 결과는 정상 출력하되 상단에 작은 회색 배지 "기본 분석 모드(상세 설명 간소화)". 사용자는 충돌 여부·순서를 그대로 받는다.
5. **같은 액티브가 여러 제품에 중복**(예: 두 세럼 다 레티놀) → 충돌 쌍은 한 번만 묶어 보여주되 "레티놀이 2개 제품(A, B)에 들어있어요 — 중복 사용 시 자극↑" 보너스 안내.
6. **한글·영문 혼용·띄어쓰기·오타** → 별칭 매칭은 공백/대소문자/하이픈 정규화 후 부분일치. Gemini 추출이 동의어를 추가로 잡는다. 매칭 못 한 토큰은 결과에 영향 없음(침묵).
7. **충돌 0건 + 안심 조합도 0건** → 초록 톤 "검사한 조합에서 알려진 충돌이 없어요. 안심하고 사용하세요." (과잉 경고 금지 원칙의 핵심 메시지).
8. **광민감 성분(레티놀/AHA/BHA) 있는데 자외선차단 미감지** → 결과 하단 보너스 노란 안내 카드: "레티놀/산 성분을 쓰는 동안엔 아침 자외선차단이 특히 중요해요." (경고 아님, 팁).
9. **매우 긴 전성분 텍스트(제품당 4000자 초과)** → 클라이언트에서 4000자로 트림 + "일부가 잘렸어요" 표시, 서버는 입력 길이 상한(zod max)으로 거부 대신 트림 후 처리.
10. **제품 6개 초과 추가 시도** → "한 번에 최대 6개까지 검사할 수 있어요" 토스트, 카드 추가 버튼 비활성.
11. **전성분에 액티브 외 자극 성분만 있는 흔한 오해 입력**(예: 향료/알코올) → 우리가 판정하지 않는 영역임을 침묵(과잉 경고 금지), 단 안심 카드 N개·경고 N개 카운트로 "검사 완료" 신뢰 신호.
12. **네트워크 단절/서버 500** → 결과 영역에 "분석에 실패했어요. 다시 시도" 버튼(입력 유지), 입력값은 로컬 상태로 보존되어 재시도 시 재입력 불필요.

### 차별화 디테일 (기본 구현 대비 "잘 만들었다")

1. **과잉 경고하지 않는 신뢰성 + 명시적 안심 카드**: 디버킹된 통념(나이아신아마이드+비타민C/레티놀, 레티놀+비타민C pH설)을 빨간 경고가 아니라 초록 "함께 써도 괜찮아요(+이유)" 카드로 적극 안심시킨다. 경쟁 도구가 "경고만" 쏟아낼 때 우리는 "괜찮은 것도 알려준다."
2. **메커니즘 기반 AM/PM 사용 순서 타임라인**: 단순 충돌 경고를 넘어 "그래서 어떻게 발라야 하나"를 아침/저녁 순서 카드로 결정론적으로 제시(화해 등이 공개적으로 약한 지점).
3. **한국 제품/한글 전성분 우선**: 별칭 사전이 한글 INCI 표기(아스코빅애씨드, 살리실릭애씨드 등)를 1순위로 포함 — 한국 화장품 뒷면을 그대로 붙여넣어도 잡힌다.
4. **LLM 없이도 동작하는 견고함**: 폴백이 "기능 저하"가 아니라 "정상 결과 + 간소한 설명"이 되도록 룰테이블에 기본 카피를 내장. 신뢰 도구로서 가용성 100%.

### "상용 제품이라면 당연히 있어야 할 것" 체크

- 결과에 대한 **근거/디스클레이머**(의료 조언 아님, 출처 톤) — **포함** (결과 하단 고정 푸터).
- **검사 결과 공유/저장**(URL이나 이미지) — **Non-goal** (로그인 없이 v1은 휘발).
- **성분 사전 검색/브라우징** — **Non-goal** (입력→결과 한 흐름만).
- **피부타입·민감도 맞춤** — **Non-goal**.
- **예시로 채우기**(빈 화면 탈출) — **포함** (입력 화면 "예시로 채우기" 버튼).
- **재검사·입력 수정 후 재실행** — **포함** (결과에서 "수정하기"로 입력 복귀, 상태 보존).

## 디자인 아이덴티티

- 레퍼런스 후보: **Toss**(해요체 마이크로카피·한 화면 한 메시지·넉넉한 여백·단일 CTA), **Stripe**(결과의 정보 정렬 — 경고 카드/순서 표의 정돈), 보조 **Family**(부드러운 radius·따뜻한 빈 상태). 형용사 3개: **신뢰감 있는 · 깨끗한 · 다정한**.
- 시각 구별 이유: 의료/안전 정보 도구라 **경고=레드 / 안심=그린 / 팁=옐로우의 시맨틱 컬러가 정보의 핵심 축**이다. 따라서 브랜드 컬러는 이 셋과 충돌하지 않는 톤(차분한 블루/슬레이트 계열)으로 잡아 "다 비슷한 그린 SaaS"와 확실히 구별되게 한다. 빨강이 무섭지 않게(자극적 순색 금지), 초록이 안도가 되게 톤을 다듬는 것이 design 스킬의 과제.

> 상세 팔레트·토큰은 builder가 design 스킬(Phase 0~1)로 확정하고 DESIGN.md에 기록한다.

## 화면 (2개)

1. **랜딩 + 입력 (한 페이지, `/`)** — 전달 메시지: "같이 발라도 되는지, 30초면 알아요."
   - 히어로: 헤드라인 + 한 줄 설명 + 우리가 검사하는 성분(레티놀·비타민C·산 등) 칩 나열.
   - 제품 입력 영역: 제품 카드 2개가 기본 노출, "+ 제품 추가"(최대 6개), 각 카드 = (선택)제품명 인풋 + 전성분 textarea + 인라인 검증 메시지.
   - "예시로 채우기" 버튼(레티놀 세럼 + 비타민C 세럼 등 시연용 프리셋), "충돌 검사하기" 단일 CTA.
   - 상태: **빈**(예시 유도) / **입력 중**(per-card 검증) / **제출 중**(CTA 로딩 스피너).
2. **결과 (`/result` 또는 동일 페이지 하단 전환)** — 전달 메시지: 한눈에 위험/안심/순서.
   - 상단 요약 배너: "경고 N개 · 안심 M개 · 분석한 제품 K개" + (폴백 시) "기본 분석 모드" 배지.
   - ① **경고 카드(레드)**: 성분 쌍 + 심각도 뱃지 + 메커니즘 + 대처법. severity 높은 순 정렬.
   - ② **안심 카드(그린)**: "함께 써도 괜찮아요" + 이유. (해당 조합 있을 때만)
   - ③ **아침/저녁 순서 타임라인**: AM/PM 두 컬럼, 각 단계 번호+성분/제형, "권장이에요" 톤 카피.
   - 보너스: 자외선차단 미감지 노란 팁 카드, 중복 액티브 안내.
   - 상태: **결과 정상** / **충돌 0건(안심 메시지)** / **액티브 미감지(중립 카드)** / **분석 실패(재시도 버튼, 입력 보존)** / **일부 제품 제외 안내**.
   - 하단 고정: 디스클레이머("일반 정보이며 의료 조언이 아니에요").

## 데이터 모델

충돌 규칙·성분 사전은 **코드 내 정적 데이터**(`src/features/checker/data.ts` 등), DB가 아니다. Supabase는 성공지표 계측용 최소 1테이블만:

- `skinclash_check_log`: `id`(uuid), `created_at`(timestamptz), `product_count`(int), `detected_active_ids`(text[]), `warn_count`(int), `safe_count`(int), `used_llm`(bool). — 전성분 원문·개인정보는 저장하지 않음(익명 집계용).

> 이 테이블 DDL은 `supabase-shared/schema.sql`에 합쳐 origin 기준으로 관리(shipping §0-1). 테이블 prefix는 `skinclash_`.

## Non-goals (이번 주에 절대 안 만드는 것)

- 바코드 스캔 / 사진 OCR 입력 (한국 커버리지 부족).
- 식약처/제품 DB 연동, 제품명으로 자동 전성분 채우기.
- 제품 추천 / 대체품 제안.
- 피부타입·민감도 진단 및 맞춤.
- 로그인 / 검사 결과 저장 / 마이페이지 / 공유 URL·이미지.
- 성분 사전 검색·브라우징 페이지.
- 충돌 여부의 LLM 판정(아키텍처 원칙상 영구 금지).
- 결제 / 프리미엄.
