# 주말마다 MVP 1개씩 찍는 공장을 만들었다 — mvp-factory를 시작한 이유

회사에서는 이미 에이전트 두 개(백엔드/프론트)를 굴리면서 개발하고 있다. 코드를 짜는 비용이 무섭게 떨어지는 걸 매일 체감하는데, 정작 퇴근하고 나면 내 개인 자산은 아무것도 쌓이지 않고 있었다. 이 간극이 계속 신경 쓰여서, 주말마다 MVP를 하나씩 찍어내는 공장 — [mvp-factory](https://github.com/mingeonho1/mvp-factory)를 만들기로 했다.

## 왜 하나에 올인하지 않고 공장인가

예전의 사이드 프로젝트는 아이디어 하나 골라서 몇 달을 갈아 넣는 방식이었다. 실패 비용이 크니까 신중할 수밖에 없었다. 그런데 에이전트한테 구현을 맡길 수 있는 지금은 MVP 하나의 비용이 주말 이틀로 줄었다. 그러면 합리적인 전략 자체가 바뀐다 ➜ 한 발의 명중이 아니라 산탄.

그래서 규칙을 이렇게 잡았다.

- 12주 동안 매주 1개, 총 12개를 배포한다
- 각 빌드는 랜딩 + 핵심 기능 1개 + 결제(또는 대기명단)까지만
- 배포 후 2주간 신호(결제/사용)가 0이면 미련 없이 아카이브
- 10개는 죽고 1~2개에서 신호가 오면 성공

12개 중 10개가 죽는 걸 전제로 깔고 시작하는 거다. 실패가 디폴트고, 신호가 예외다. 개인적으로 느끼기에 이 전제를 받아들이는 순간 사이드 프로젝트의 심리적 무게가 확 줄었다 (몇 달 갈아 넣은 프로젝트를 접는 건 아프지만, 주말 하나 접는 건 그냥 다음 주가 있을 뿐이라..).

## 공장의 구조 — 에이전트 3개, 훅 3개

매주 같은 작업을 반복하려면 하네스가 먼저다. 템플릿 리포 하나를 만들고, 매주 "Use this template"으로 복제해서 시작한다.

에이전트는 세 명으로 분리했다.

- **lead** — 기획만 한다. PLAN.md가 유일한 산출물이고, 코드는 못 쓰게 막아놨다
- **builder** — 구현만 한다. 태스크마다 lint + typecheck + test 통과가 완료 조건
- **reviewer** — 검수만 한다. 체크리스트 10개에 PASS/FAIL을 파일:라인 근거와 함께 판정

처음엔 리더(기획+검수) / 개발자 2인 체제를 생각했는데, 기획한 에이전트가 자기 계획을 검수하면 같은 맹점을 두 번 통과한다는 문제가 있었다. 검수는 그 계획을 처음 보는 깨끗한 컨텍스트가 해야 잡는다 ➜ 그래서 3인.

그리고 규칙 강제는 프롬프트가 아니라 훅(hook)으로 했다. "꼭 lint 돌려줘"라고 부탁하는 건 모델의 해석에 의존하지만, 훅은 결정론적으로 무조건 실행된다.

```json
"Stop": [
  {
    "hooks": [
      { "type": "command", "command": "bash .claude/hooks/verify.sh" }
    ]
  }
]
// ...중략
```

- 세션 종료 시 verify.sh가 `pnpm check`를 돌리고, 실패하면 exit 2로 종료 자체를 차단한다
- 위험 명령(rm -rf, force push, .env 출력)은 guard.sh가 실행 전에 막는다
- "통과 못 하면 끝낼 수 없다"가 시스템 레벨에서 보장되는 셈이다

여기에 eslint로 파일 200줄, 함수 40줄, 중첩 3단계 초과를 에러로 박아놔서 코드가 비대해지는 것도 숫자로 막았다.

## 모델 배치 — 비싼 지능은 양 끝에만

토큰의 80%는 builder의 구현 루프(읽기 ➜ 수정 ➜ check ➜ 재수정)에서 난다. 그래서 builder만 Sonnet으로 내리고 lead와 reviewer는 상위 모델을 유지했다. 명세는 위에서 빡빡하게, 구현은 싸게, 검증은 다시 위에서. 샌드위치 구조다.

대신 Sonnet이 헤매지 않도록 lead의 태스크 작성 기준을 강제했다. 모든 태스크는 완료 조건 / 대상 경로 / 참조 문서 / 테스트 매핑 4요소를 채워야 한다. "동작한다" 같은 완료 조건은 금지고, "X를 입력하면 Y가 표시된다"처럼 검증 가능한 문장이어야 한다. 명세 품질이 곧 구현 품질이라는 걸 받아들이고 나니, 비싼 모델로 의사코드까지 써주는 건 오히려 낭비라는 결론이 나왔다 (어떻게 만들지는 builder의 영역으로 남겨야 더 잘 만든다).

## 우당탕탕 셋업 — 1년 전 지식은 전부 틀렸다

하네스 셋업에서 제일 크게 데인 건, 작년 지식이 거의 다 깨져 있었다는 점이다.

- **pnpm 10 ➜ 11**: 빌드 스크립트 승인 설정인 `onlyBuiltDependencies`가 제거되고 `allowBuilds` 맵으로 바뀌었다. package.json의 pnpm 필드 자체를 더 이상 안 읽는다. 같은 에러를 세 번 보고서야 공식 문서를 다시 팠다...
- **Next 16**: eslint 설정에서 FlatCompat 레이어를 걷어내고 `eslint-config-next/core-web-vitals`를 직접 import하는 방식으로 바뀌었다. 구방식으로 쓰면 순환 구조 JSON 에러가 터진다
- **git 계정 분기**: 회사/개인 리포가 같은 폴더에 섞여 있어서 gitdir 분기가 안 맞았다. `includeIf "hasconfig:remote.*.url:..."`로 리모트 주소 기준 분기로 바꾸니 리포를 몇 개 파든 자동으로 정리됐다

AI한테 "최신으로 해줘"라고 시켜도 학습 시점의 지식으로 만들기 때문에, 버전 민감한 설정은 공식 문서를 같이 들이밀어야 한다는 걸 다시 확인했다.
